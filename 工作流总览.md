# V7ex 工作流总览

## 1. 系统总览
- 前端：Vue3 + Pinia + Vue Router + Element Plus
- 后端：FastAPI
- 数据层：MySQL（SQLAlchemy + pandas）
- 核心目标：围绕“合同/订单/库存/入库/发货/档案”形成闭环

---

## 2. 端到端主流程图

## 3. 角色泳道流程

### 3.1 Sales（销售）
1. 登录系统
2. 在“销售下单”创建订单（可手工或由合同转换）
3. 在“订单配货”选择机台并执行配货
4. 跟踪订单发货状态

### 3.2 Prod / Inbound（生产/入库）
1. 登录系统
2. 在“成品入库”上传跟踪单或自动生成待入库记录
3. 审核并确认导入
4. 在库位维度执行入库（入库到具体 `Location_Code`）
5. 必要时执行机台编辑、发货撤回等操作

### 3.3 Boss / Admin（管理）
1. 进行生产统筹与合同状态管理
2. 审核用户注册申请与角色
3. 在“发货复核”执行正式发货
4. 通过“交易日志”追踪关键操作

## 4. 核心状态流转

### 4.1 库存机台状态
1. `待入库`
2. `库存中（库位）`
3. `待发货`
4. `已出库`
5. 异常回退：`待发货/已出库` 可按业务规则回退到 `待入库`

### 4.2 订单状态
1. `active`（默认有效）
2. `done`（后端按发货数量对齐）
3. `deleted`（前端管理页逻辑删除）

### 4.3 合同状态（典型）
1. `待规划`
2. `已规划`
3. `已下单`
4. `已转订单`
5. `已取消`（按业务操作）

## 5. API 对照表（高频）

### 5.1 认证与用户
- `POST /api/v1/auth/login`：登录，返回 JWT + 用户信息
- `POST /api/v1/users/register`：用户注册申请
- `POST /api/v1/users/audit`：审核通过/拒绝
- `PATCH /api/v1/users/{username}`：用户角色/状态/姓名更新

### 5.2 统筹与订单
- `GET /api/v1/planning/`：读取统筹/合同数据
- `GET /api/v1/planning/orders`：读取订单
- `POST /api/v1/planning/orders`：创建订单
- `PUT /api/v1/planning/orders/{order_id}`：编辑订单
- `POST /api/v1/planning/orders/{order_id}/allocate`：配货锁定
- `POST /api/v1/planning/orders/{order_id}/release`：释放配货
- `POST /api/v1/planning/contract/{contract_id}/status`：合同状态更新
- `POST /api/v1/planning/contract/{contract_id}/save-plan`：保存规划
- `POST /api/v1/planning/contract/{contract_id}/link-order`：合同关联订单

### 5.3 库存、入库、发货
- `GET /api/v1/inventory/`：读取库存
- `POST /api/v1/inventory/import-staging/upload`：上传跟踪单并解析
- `POST /api/v1/inventory/import-staging/save`：保存待入库清单
- `POST /api/v1/inventory/import-staging/import-confirm`：确认导入库存
- `POST /api/v1/inventory/import-staging/auto-generate`：自动生成待入库数据
- `POST /api/v1/inventory/inbound-to-slot`：按库位入库/调拨
- `GET /api/v1/inventory/shipping/pending`：读取待发货
- `POST /api/v1/inventory/shipping/confirm`：正式发货
- `POST /api/v1/inventory/shipping/revert`：发货撤回

### 5.4 档案与日志
- `GET /api/v1/inventory/machine-archive/serials`：流水号列表
- `POST /api/v1/inventory/machine-archive/{serial_no}/upload`：上传档案
- `GET /api/v1/inventory/machine-archive/{serial_no}/files`：文件列表
- `DELETE /api/v1/inventory/machine-archive/{serial_no}/files/{file_name}`：删除档案
- `GET /api/v1/logs/transactions`：交易日志查询

## 6. 关键技术机制

### 6.1 前端鉴权
1. 请求拦截器自动附带 `Authorization: Bearer <token>`
2. 路由守卫控制未登录跳转与角色访问
3. `401` 统一触发登出并跳转登录页

### 6.2 数据一致性
1. 入库、配货、发货等关键动作落库并写交易日志
2. 订单与库存通过 `占用订单号` 字段关联
3. 部分流程（如入库到库位）包含事务与容量校验

### 6.3 性能策略
1. 前端路由懒加载
2. Pinia 内存短 TTL 缓存
3. 大列表页面采用筛选索引/虚拟列表优化

---

## 7. 异常与回滚分支
1. 登录失败：前端展示后端 detail，保留当前页
2. token 失效：统一 `401` -> 自动登出 -> 回登录页
3. 配货超量/机型不匹配：前端先拦截，后端再校验
4. 库位满载/锁定：后端拒绝入库并返回错误码
5. 发货撤回：按流水号批量回退到 `待入库`

## 8. 发布检查建议（精简版）
1. 后端健康检查：`GET /health` 返回 `200`
2. 前端构建可用：`frontend/dist/index.html` 存在
3. 核心路径回归：登录 -> 下单 -> 配货 -> 入库 -> 发货 -> 日志
4. 浏览器覆盖：Chrome / Edge / Firefox

## 9. 当前已识别风险点
1. 注册页可选 `Inbound`，但后端注册模型枚举未覆盖 `Inbound`（会导致该角色注册失败）
2. 部分业务 API 鉴权主要依赖前端路由守卫，后端接口级权限收敛仍可加强
