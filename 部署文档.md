# V7ex 部署文档（第一版）

## 1. 环境要求
- 操作系统：Windows Server / Linux（推荐 Ubuntu 22.04+）
- Node.js：`18+`
- npm：`9+`
- Python：`3.10+`
- 数据库：SQLite（当前默认）或后续替换为 MySQL/PostgreSQL

## 2. 目录说明
- 后端根目录：`V7ex/`
- 前端目录：`V7ex/frontend/`
- 前端构建产物：`V7ex/frontend/dist/`

## 3. 后端部署
### 3.1 配置环境变量
- 复制模板：
```bash
cp .env.backend.example .env
```
- 重点检查：
  - `UVICORN_RELOAD=false`（生产环境）
  - `UVICORN_WORKERS`（建议 >= 2）
  - `UVICORN_LOG_LEVEL=info`
  - `ADMIN_PASSWORD`
  - `MYSQL_*`（若启用 MySQL）

### 3.2 安装依赖
```bash
pip install -r requirements.txt
```

### 3.3 启动后端（开发/测试）
```bash
python run_api.py
```

### 3.4 生产启动建议（Windows）
```bash
scripts\start_backend_prod.bat
```
- 建议通过 NSSM / WinSW 注册为系统服务。
- 日志建议重定向到文件并按天切分。
- 启动后可执行健康检查：
```bash
python scripts/check_health.py --url http://127.0.0.1:8000/health
```

### 3.5 服务化部署模板（Windows）
- NSSM 示例脚本：
```bash
scripts\install_nssm_example.bat
```
- WinSW 配置模板：
```text
scripts\winsw\V7exBackend.xml.example
```

### 3.6 日志轮转建议
- 后端日志建议按“天 + 大小”滚动，至少保留 7~14 天。
- 若使用 NSSM，建议输出到 `logs/backend.out.log` 与 `logs/backend.err.log` 并交给系统计划任务轮转。
- 若使用 WinSW，可直接使用模板中的 `<log mode="roll-by-size-time">`。

## 4. 前端部署
### 4.1 安装依赖
```bash
cd frontend
npm install
```

### 4.2 配置环境变量
- 复制模板：
```bash
copy .env.production.example .env.production
```
- 按实际网关地址修改：`VITE_API_BASE_URL`

### 4.3 构建
```bash
npm run build
```

### 4.4 本地预览
```bash
npm run preview
```

### 4.5 Nginx 静态托管（示例）
```nginx
server {
  listen 80;
  server_name _;

  root /path/to/V7ex/frontend/dist;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  location /api/ {
    proxy_pass http://127.0.0.1:8000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

## 5. 发布流程（建议）
1. 拉取最新代码并备份当前版本。
2. 执行前端构建：`npm run build`。
3. 生成性能基线：`scripts\perf_baseline.bat`（输出 `frontend/dist/perf-baseline.md`）。
4. 执行后端启动：`scripts\start_backend_prod.bat`。
5. 执行健康检查：`python scripts/check_health.py`。
6. 执行上线前预检：`scripts\preflight_release.bat`。
7. 发布 `dist` 到静态目录，重载 Nginx。
8. 回归验证：登录、权限、库位大屏全屏、下单/配货/入库主流程。

## 6. 回滚方案
1. 回滚到上一个 `dist` 目录备份版本。
2. 回滚后端代码到上一个稳定版本。
3. 重启后端服务并刷新 Nginx。
4. 执行最小回归（登录 + 核心流程）。

## 9. 已知说明

## 10. 性能基线产物
- 生成命令：`scripts\perf_baseline.bat`
- 默认输出：`frontend/dist/perf-baseline.md`
- 用途：记录每次发布的产物体积变化，作为“性能基准测试”追踪依据。
- 对比命令：`scripts\perf_compare.bat --previous <上一次基线文件>`
- 对比输出：`frontend/dist/perf-compare.md`
