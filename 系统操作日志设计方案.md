# 系统操作日志设计方案

本方案中的操作日志，定位为**面向业务管理者的管理日志**，重点用于回答以下问题：

- 谁做了这件事
- 在什么时间做的
- 在哪个业务模块做的
- 做了什么操作
- 这次操作的业务内容是什么

因此，该日志表不追求记录过多技术细节，而是强调信息简洁、可读、方便业务管理者查询。

## 1. 数据库表结构设计 (`sys_operation_log`)

建议将操作日志与现有业务流水日志分离，单独建立一张 `sys_operation_log` 表，仅保存业务管理者真正关心的信息。

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | BigInt | 主键 |
| `user_id` | String/Int | 操作人 ID |
| `username` | String | 操作人名称 |
| `operate_time` | DateTime | 操作时间 |
| `module` | String | 所属模块，如订单管理、用户管理 |
| `action_type` | String | 操作类型，如新增、修改、删除、审核、导出、登录 |
| `biz_type` | String | 业务对象类型，如订单、用户、角色、付款单 |
| `content` | String/Text | 操作内容摘要，使用业务语言描述 |

### 字段设计说明

- `user_id` 和 `username` 用于明确是哪位人员进行了操作。
- `operate_time` 用于记录业务动作发生的具体时间。
- `module` 用于区分本次操作属于哪个业务模块。
- `action_type` 用于归类本次动作的类型，便于后续统计。
- `biz_type` 用于说明本次操作的业务对象。
- `content` 是最关键的字段，建议直接写成业务管理者能看懂的一句话。

### `content` 示例

- `创建订单：A20260418001`
- `修改客户信息：深圳华南贸易有限公司`
- `禁用用户：张三`
- `审核通过付款单：FK20260418009`

## 2. 字段获取方式

这张表中的字段，不建议全部依赖前端传值，而应由后端按职责分层获取。

| 字段名 | 获取方式 | 建议来源 |
| :--- | :--- | :--- |
| `id` | 系统自动生成 | 数据库自增或后端统一生成 |
| `user_id` | 登录态获取 | 从当前登录用户信息中获取 |
| `username` | 登录态获取 | 从当前登录用户信息中获取 |
| `operate_time` | 系统自动生成 | 后端写入当前时间 |
| `module` | 接口固定声明 | 在路由或装饰器中定义 |
| `action_type` | 接口固定声明 | 在路由或装饰器中定义 |
| `biz_type` | 接口固定声明 | 在路由或装饰器中定义 |
| `content` | 业务逻辑拼接 | 在服务层或接口中生成 |

### 获取原则

1. `id` 和 `operate_time` 由后端统一生成，不依赖前端传入。
2. `user_id` 和 `username` 从当前登录态中获取，不从请求参数中读取。
3. `module`、`action_type`、`biz_type` 在接口开发时固定声明，不建议程序自动猜测。
4. `content` 由具体业务逻辑生成，要求使用业务管理者容易理解的描述方式。

---

## 3. 推荐实现方式

对于 FastAPI 项目，推荐采用 **装饰器 + 异步写入** 的方式实现。

### 推荐原因

- 装饰器适合给关键业务接口打上固定标签，例如模块、操作类型、业务对象类型。
- 用户信息和时间可以在请求处理过程中自动获取。
- `content` 可以在业务处理成功后，由代码拼接为一条可读的业务描述。
- 异步写入不会阻塞主业务接口响应。

### 适合放在装饰器中的字段

- `module`
- `action_type`
- `biz_type`

### 适合运行时自动获取的字段

- `user_id`
- `username`
- `operate_time`

### 适合由业务逻辑生成的字段

- `content`

## 4. 关键代码示例

下面提供一个更贴近业务管理视角的示例。核心思路是：

- 在装饰器中固定声明模块、操作类型、业务对象类型
- 在请求上下文中获取当前登录用户
- 在业务执行成功后，拼接 `content`
- 通过后台任务异步写入日志表

```python
from functools import wraps
from datetime import datetime
from fastapi import BackgroundTasks, Request


def async_save_log(log_data: dict):
    # 在此处将 log_data 写入 sys_operation_log 表
    pass


def record_log(module: str, action_type: str, biz_type: str):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, request: Request, background_tasks: BackgroundTasks, **kwargs):
            user = getattr(request.state, "user", {}) or {}

            result = await func(*args, request=request, background_tasks=background_tasks, **kwargs)

            log_data = {
                "user_id": user.get("user_id"),
                "username": user.get("username", "Unknown"),
                "operate_time": datetime.now(),
                "module": module,
                "action_type": action_type,
                "biz_type": biz_type,
                "content": result.get("log_content", ""),
            }

            background_tasks.add_task(async_save_log, log_data)
            return result

        return wrapper
    return decorator
```

**路由使用示例：**

```python
@router.post("/create")
@record_log(module="订单管理", action_type="新增", biz_type="订单")
async def create_order(request: Request, background_tasks: BackgroundTasks, payload: dict):
    order_code = "A20260418001"

    # 真实业务处理

    return {
        "msg": "success",
        "log_content": f"创建订单：{order_code}",
    }
```

该实现方式下，日志表中的 `content` 字段不再保存技术参数，而是直接保存业务可读内容，更符合管理者查看习惯。

## 5. 使用建议

1. 该表用于展示“业务管理动作”，不建议继续加入 IP、URL、请求参数、异常堆栈等开发侧字段。
2. `content` 应尽量使用自然语言描述，避免只写内部编号或技术字段名。
3. 建议只记录关键业务动作，例如新增、修改、删除、审核、导出、登录，不必覆盖所有查询接口。
4. 如果后续需要更强的技术审计能力，可以在此表之外另建审计日志表，与本表分开维护。

---

## 6. 结论

`sys_operation_log` 应定位为一张**面向业务管理者的操作日志表**，重点记录“谁、何时、在哪个模块、对什么业务对象、做了什么”。  

这样的设计有以下优点：

- 字段更少，页面展示更清晰
- 日志内容更容易被业务人员理解
- 后续做列表查询、按模块统计、按操作类型统计都更方便
- 与技术审计日志职责分离，避免表结构越来越复杂
